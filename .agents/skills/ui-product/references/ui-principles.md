# UI Principles

## 기본 방향

- 상위 레벨에서는 블록 구분감이 명확해야 합니다.
- Markdown 블록 안에서는 직접 편집감이 좋아야 합니다.
- 제품은 단순하고 빠르게 느껴져야 합니다.

## 현재 제품 해석

- “Notion식”은 Markdown이 렌더링된 것처럼 보이면서도 바로 수정 가능한 경험을 뜻합니다.
- “Heynote식”은 상위 블록이 분명히 분리되어 있고, 블록 단위 사고가 유지되는 경험을 뜻합니다.
- 둘을 섞을 때 어떤 편집기 구조가 맞는지는 아직 열어 둡니다. 다만 상위 블록 구분감이 유지되는지가 더 중요합니다.

## 선호하는 UX 방향

- Markdown 체크리스트, 헤딩, 리스트 같은 구조가 자연스럽게 보이고 수정 가능해야 합니다.
- 상위 블록 경계는 시각적으로 분명해야 합니다.
- 별도 preview pane 없이도 읽기와 쓰기가 모두 가능해야 합니다.
- 무거운 패널, 과한 설정, 복잡한 툴바는 기본값으로 두지 않습니다.
- `Cmd+A`는 현재 블록 전체 선택, 한 번 더 누르면 모든 블록 전체 선택으로 동작합니다.
- `Cmd+S`는 저장/flush를 명시적으로 실행할 수 있어야 합니다.
- 디자인은 macOS 스타일을 대체로 준수하고, 다크/라이트 모드를 명확히 구분합니다.
- 사용자가 언급한 macOS 26 liquid 계열 감성을 참고하되, 과한 효과보다 안정적인 읽기/편집감을 우선합니다.

## 미정인 항목

- 문서 전체를 하나의 큰 editor로 구성할지 여부는 아직 결정하지 않았습니다.

## 금지 패턴

- 블록 경계를 희석시키는 UI
- 미리보기와 원문 편집을 계속 왔다 갔다 해야 하는 흐름
- 제품 방향을 흐리는 과한 기능 추가
- 불필요한 저장 버튼 중심 UX
- 삭제 때마다 흐름을 끊는 과한 확인 모달
- 사용자 승인 없이 UI 철학 문서를 확장하거나 재정의하는 작업

## Gotchas

- “심플함”을 기능 부족으로 오해하지 않습니다.
- “직접 편집”을 위해 반드시 전형적인 WYSIWYG 에디터 전체 구조로 가야 한다고 가정하지 않습니다.
- UI 편의 기능을 추가하더라도 블록 정체성을 약화시키면 실패로 봅니다.
