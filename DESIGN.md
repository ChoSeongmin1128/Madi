# Madi Design

이 문서는 Madi의 UI/UX 작업 기준입니다. 새 화면, 설정, 컨트롤, 상태 표현, 색상 조정을 할 때는 이 문서를 먼저 확인합니다.

## 제품 정체성

Madi는 `컴팩트한 macOS 블록 노트`입니다.

- macOS 단독 데스크톱 앱처럼 자연스럽고 조용해야 합니다.
- 상위 구조는 Heynote처럼 블록 단위 사고가 분명해야 합니다.
- Markdown 블록 안에서는 Notion처럼 읽기와 쓰기가 이어지는 직접 편집감을 지향합니다.
- 빠른 메모, 반복 사용, 키보드 중심 조작을 방해하지 않아야 합니다.
- 기능을 많이 보여주는 앱보다 매일 열어 둬도 거슬리지 않는 작업 도구에 가깝게 만듭니다.

## 시각 방향

Apple 디자인은 `네이티브 macOS 앱 감각`으로 해석합니다.

- SF Pro 계열 타이포그래피, 시스템 컬러, 네이티브 컨트롤 감각을 우선합니다.
- Liquid/Glass 효과는 보조 표면감으로만 사용합니다.
- 반투명, blur, 광택은 정보 구조를 흐리면 줄입니다.
- Apple 제품 페이지식 넓은 여백, 대형 히어로, 이미지 중심 구성은 적용하지 않습니다.
- Raycast처럼 빠르고 도구적인 조작감은 참고하되, 명령 팔레트 앱처럼 과도하게 차갑게 만들지 않습니다.
- Notion처럼 편집 표면은 차분하게 유지하되, 블록 구분감이 사라지면 실패로 봅니다.

## 밀도와 레이아웃

Madi는 넓고 프리미엄한 페이지보다 컴팩트하고 읽기 쉬운 앱이어야 합니다.

- 화면당 정보 밀도는 유지합니다.
- 여백은 장식이 아니라 그룹 구분, 터치/클릭 안정성, 가독성을 위해 씁니다.
- 설정, 사이드바, 메뉴는 조밀하지만 답답하지 않아야 합니다.
- 카드 안에 카드를 중첩하지 않습니다.
- 큰 제목과 넓은 패딩은 문서 제목처럼 위계가 큰 영역에만 씁니다.
- 반복 항목은 높이, radius, padding이 안정적이어야 하며 hover나 상태 텍스트 때문에 흔들리지 않아야 합니다.
- 모바일 폭에서는 텍스트가 버튼이나 칩 밖으로 넘치지 않아야 합니다.

## 색상과 표면

현재 인디고 계열은 Madi의 포커스/선택 포인트로 유지합니다.

- 인디고는 선택, 포커스, 업데이트 CTA, 중요한 상태 강조에 제한적으로 사용합니다.
- 일반 배경과 패널은 시스템 배경에 가까운 중립 톤을 기본으로 합니다.
- semantic color는 명확한 상태에만 씁니다: error, warning, success, progress.
- danger는 삭제, 덮어쓰기, 되돌리기 어려운 복구 작업에만 씁니다.
- 다크 모드는 단순 반전이 아니라 대비, glow, border가 과해지지 않도록 별도로 조정합니다.
- 한 화면이 보라/파랑 계열만으로 읽히면 색상 균형을 다시 잡습니다.

## 블록 시스템

블록 구분은 Madi의 정체성입니다.

- 은은한 교차 틴트로 인접 블록의 배경 톤 차이를 유지합니다.
- 틴트는 문서 읽기를 방해하지 않을 정도로 낮은 채도와 낮은 대비를 사용합니다.
- Markdown/Text/Code 종류별 색상 차이보다 문서 안의 블록 경계를 우선합니다.
- 선택된 블록은 전체 border와 glow로 명확히 표시합니다.
- 텍스트 선택, 블록 선택, 전체 블록 선택은 서로 구분돼야 합니다.
- 드래그 핸들, 블록 메뉴, 종류 배지는 기본 상태에서는 조용하고 hover/focus/selection에서 드러납니다.
- 블록 구분을 없애는 큰 단일 에디터형 표면은 기본 방향이 아닙니다.

## 설정 UX

설정의 장기 목표는 `네이티브 설정창형` 구조입니다.

- 단일 긴 스크롤 모달보다 사이드 카테고리와 상세 패널을 가진 설정창을 지향합니다.
- 기본 카테고리는 일반, 외형, 편집, 창, 동기화, 업데이트, 고급으로 나눕니다.
- 사용자가 자주 바꾸는 설정은 앞쪽에 둡니다.
- 위험하거나 복구용인 작업은 고급 또는 위험 영역으로 격리합니다.
- 내부 구현 용어는 기본 설정 화면에 노출하지 않습니다.
- 진단 정보는 고급 영역에서만 보여주고, 사용자가 판단 가능한 짧은 설명을 붙입니다.
- 설정 문구는 기능 이름보다 사용자가 얻게 되는 결과를 우선합니다.

## 컨트롤 사용 기준

컨트롤은 의미별로 일관되게 씁니다.

- Switch/checkbox: 켜짐/꺼짐처럼 즉시 적용되는 binary 설정.
- Segmented control: 2-5개 정도의 상호 배타적 mode 선택.
- Button: 명령 실행, 새로고침, 복구, 삭제, 업데이트 적용.
- Icon button: 사이드바 열기, 닫기, 메뉴, 복사, 삭제처럼 자주 쓰는 도구성 액션.
- Slider: 투명도, 글자 크기처럼 연속적인 값.
- Text field: 검색, 제목, 단축키 캡처처럼 입력 목적이 분명한 경우.
- Status chip: 저장, 동기화, 업데이트처럼 현재 상태를 짧게 표시하는 경우.

컨트롤 라벨은 짧게 씁니다. 설명이 필요하면 보조 문구로 분리합니다.

## 상태와 문구

상태 문구는 짧고 판단 가능해야 합니다.

- 좋은 예: `저장됨`, `저장 중`, `동기화 기록 없음`, `업로드 대기 2건`, `업데이트 준비됨`
- 피할 예: `대기 중`, `처리 중`, `Cloud에 다시 업로드`, `Raw Pending`, `Backoff`
- iCloud나 업데이트처럼 성공 여부가 중요한 기능은 마지막 확인 시점이나 다음 행동을 드러냅니다.
- 오류 문구는 사용자가 할 수 있는 다음 행동이 있으면 함께 제시합니다.
- 개발자/내부 진단 용어는 사용자 기본 흐름에 노출하지 않습니다.

## 금지 패턴

- 블록 경계를 흐리는 전체 단일 문서 표면.
- preview pane과 원문 편집을 오가야 하는 Markdown 흐름.
- Apple 마케팅 페이지처럼 넓은 히어로, 대형 이미지, 과한 여백을 앱 내부 UI에 적용하는 방식.
- 반투명과 blur가 글자 대비를 떨어뜨리는 표면.
- 같은 의미의 설정을 segmented, checkbox, button으로 섞어 표현하는 방식.
- 설정 기본 화면에 진단 필드, bridge, token, queue 같은 내부 용어를 직접 노출하는 방식.
- 단순함을 이유로 필요한 상태 피드백을 숨기는 방식.
- 삭제와 복구처럼 위험도가 다른 명령을 같은 시각 강도로 배치하는 방식.

## AI 작업 기준

AI 에이전트가 UI/UX를 수정할 때는 다음 순서로 판단합니다.

1. 블록 구분감이 유지되는가.
2. macOS 앱처럼 자연스럽고 컴팩트한가.
3. 사용자가 현재 상태와 다음 행동을 이해할 수 있는가.
4. 컨트롤 의미가 일관되는가.
5. 다크/라이트 모드에서 대비와 표면감이 모두 안정적인가.
6. 텍스트, 버튼, 칩, 메뉴가 좁은 폭에서도 겹치거나 넘치지 않는가.

구현 변경이 이 문서와 충돌하면 디자인 문서를 먼저 갱신하지 말고, 충돌 지점을 사용자에게 설명하고 방향을 확인합니다.
